## 1. Code→Spec Conversion: Concrete Implementations

### 1.1 Reverse Requirements Engineering — Use Case Recovery

The field has no unified product, but there are active research systems:

**UCRBench (Dec 2025)** — arXiv:2512.13360 — first systematic benchmark for *use case recovery* from code. Nine real-world Java projects (863–66,609 LOC), 232 user-goal use cases, 324 subfunction use cases. Four LLMs tested: GPT-5, GPT-5-mini, DeepSeek-V3.2 (chat + reasoning). Results:

| Metric | Best result |
|--------|-------------|
| Actor accuracy | 80.6% (DeepSeek, user goals) |
| Name accuracy | 35–43% |
| Path accuracy | 79–80% (DeepSeek, subfunctions) |
| Omission rate | **49–64% for user-goal cases** |

The omission rate is the critical number: LLMs miss 49–64% of user-level goals. Key failure modes:
- **Granularity explosion**: GPT generated 2,103 use cases for one project (vs. ~30 actual)
- **Actor confusion**: cannot distinguish user-level vs developer-level actors, fails on domain-specific roles (student, passenger, admin)
- **Abstraction inconsistency**: merging subfunctions into user goals loses coherence in multi-module systems
- **Domain opacity**: professional domains require regulatory awareness that LLMs lack

**Practical finding**: current LLMs can partially recover *what code does* at subfunction level (80% path accuracy) but fail at *why it exists* at user-goal level (49% omission). The gap is widest precisely where spec value is highest.

**SBC Metric (Feb 2025)** — arXiv:2502.07835 — "Semantic-BLEU-Completeness" score for reverse-generated requirements. The reverse generation technique works backward: given AI-generated code, prompt an LLM to reconstruct requirements, then compare to the original spec using three sub-scores:
- Semantic similarity (embedding-based)
- BLEU (token-level)
- Completeness analysis (what original requirements are missing from reconstruction)

The completeness sub-score is novel: it highlights *missing features and hallucinations* rather than just measuring similarity. Use case: given a PR with LLM-generated code, developers can run SBC to see what requirements the code fails to implement and what hallucinated behavior exists.

**Grammar Inference via Symbolic Parsing (Mar 2025)** — arXiv:2503.08486 — a different angle: instead of reconstructing NL requirements, extract the *implicit grammar* encoded in parser code. STALAGMITE analyzes recursive descent parsers statically:
1. Associate each `parse_ELEM()` function with a grammar nonterminal `<ELEM>`
2. Limit recursion/loops to enumerate finite execution paths
3. Each path → one grammar expansion alternative

Accuracy: 99–100% on TINY-C and JSON parsers. This is effectively extracting the behavioral specification of what inputs a system accepts — a form of *interface spec extraction* rather than intent extraction. Directly applicable when the spec needed is "what does this accept/reject?"

### 1.2 Architecture Documentation Generation from Code

**Automated SAD generation (Nov 2025)** — arXiv:2511.05165 — semi-automatic Software Architecture Description generation combining reverse engineering + LLMs for C++ systems. Two views generated:

- **Static view**: extract component diagram → filter architecturally significant elements via prompt engineering (reduces noise from utility/helper code)
- **Behavioral view**: state machine diagrams generated by analyzing code logic with few-shot prompting

Key technique: few-shot prompting with domain-specific examples dramatically improves behavioral view quality. Without domain examples, state machine diagrams are wrong or incomplete.

**Architecture-Code Traceability (Nov 2025)** — arXiv:2511.02434 — ExArch + ArTEMiS systems:
- ExArch: F1=0.86, extracts component names from Software Architecture Documentation + source code without manual models
- ArTEMiS: F1=0.81, matches architectural entities from documentation to code

Both systems work best when documentation *exists* to cross-reference. For repos with no architecture docs, performance degrades significantly.

**Gap**: no production tool generates arc42 or C4 model diagrams automatically from code. SPViz (arXiv:2401.17063) offers a DSL-driven approach where architects define their own visualization tool, which then automatically updates as the project changes — closest to what's needed, but requires initial manual DSL definition.

### 1.3 Automated Refactoring Systems and Intermediate Representations

**Sourcery.ai** (commercial) uses a combination of static analysis and pattern-based transformations. Their blog describes eight categories of Python refactoring patterns. The system does NOT use an IR like LLVM — it operates at the AST level with pattern templates, which preserves semantic context. This is consistent with the LLVM IR finding: AST-level operations are less lossy than binary-level IR.

**Differential Fuzzing for Refactoring Verification (Feb 2026)** — arXiv:2602.15761 — uses Google's Atheris (coverage-guided Python fuzzer) to verify that LLM-generated refactorings preserve functional equivalence:

*Process*:
1. Infer input constraints using GPT-4 (input types, inter-dependencies, relational conditions)
2. Generate 1,000–2,000 test inputs satisfying constraints
3. Execute both original and refactored code on all inputs
4. Binary equivalence: any single output mismatch → non-equivalent

*Results across 6 LLMs*:

| Model | Non-equivalent refactorings |
|-------|----------------------------|
| Codestral | 35% |
| StarChat2 | 34% |
| CodeLlama | 26% |
| Qwen-2.5 | 22% |
| Olmo-3 | 22% |
| GPT-4o | 19% |

**Critical finding**: 21% of non-equivalent refactorings pass existing test suites — they are undetected bugs. Even the best LLM (GPT-4o) produces 19% semantically wrong refactorings on real code.

**Anka DSL (Dec 2025)** — arXiv:2512.23214 — constrained DSL for LLM code generation in data pipelines. Constrained syntax eliminates the ambiguity problem: "errors stem from the flexibility of general-purpose languages, which permits multiple valid approaches." Results on 100 multi-step pipeline tasks:
- Claude 3.5 Haiku: Anka=100% vs Python=60% (+40pp)
- GPT-4o-mini: +26.7pp on multi-step tasks

**Implication**: the LLM→code generation failure rate is a function of target language flexibility. A spec language that constrains the target (like CodeSpeak's minimal NL → specific code) should inherently produce fewer semantic errors than asking LLMs to generate Python.

## 2. Behavioral Equivalence: Production Systems

### 2.1 Google's Large-Scale Code Migration

**"What a diff makes" (Nov 2025)** — arXiv:2511.00160 — AIMigrate, an open-source Python tool for dependency-version migration using diff-context prompting. Core finding: providing *diffs* (what changed between library versions) outperforms providing full code files for migration tasks. 65% correct in single run, 80% with multiple runs on TYPHOIDSIM migration. The diff functions as a compressed, change-focused spec — exactly the "spec delta → code delta" problem that CodeSpeak identifies as unsolved.

**Google Kythe Migration (2025)** — arXiv:2504.09691 — 39 distinct ID-field migrations over 12 months by 3 developers across 149 teams. Fine-tuned Gemini model. Technical detail:

*Change location discovery*:
```
Kythe code index → direct references → indirect references (up to depth 5) → superset of candidate locations
```

*Categorization* (4 classes):
- `not-migrated` (100% confidence, regex-based)
- `irrelevant` (confirmed unrelated)
- `relevant` (needs LLM change)
- `left-over` (manual review required)

*Validation pipeline* (6 sequential checks):
1. Successful completion
2. Whitespace-only change detection
3. AST parser validity
4. LLM "punt" detection (model refused to change)
5. Build compilation
6. Regression test passage

74.45% of 93,574 total edits generated by LLM. Estimated 50% time reduction. Languages: Java, C++, Python (Dart poorly supported).

**Critical limitation**: NO formal equivalence verification. Behavioral equivalence is asserted by test passage only. The system explicitly relies on code owner test suites — reinforcing the test-coverage-is-the-oracle problem.

**Environment-in-the-Loop (Feb 2026)** — arXiv:2602.09944 — argues that code migration automation is incomplete without co-migrating the runtime environment. "Code and its environment are intricately intertwined." No experimental results yet — a framework proposal — but the insight is valid: migrating a library API call also requires migrating config files, environment variables, deployment scripts. Spec-to-code systems face the same problem: a spec that governs business logic doesn't govern infrastructure.

### 2.2 AWS Cedar Formal Verification — Complete Details

**Cedar Verification-Guided Development (VGD)** — arXiv:2407.01688 — the most complete production formal verification case study available:

**Tool**: Lean 4 (the Lean theorem prover). Cedar's evaluator, validator, and symbolic compiler written as recursive Lean functions.

**Code metrics**:
```
Lean model:   1,673 LOC
Lean proofs:  5,714 LOC  (3.4x proof-to-model ratio)
Rust prod:   15,693 LOC  (10x larger than Lean model)
Rust tests:  20,458 LOC
```

**7 properties proven** in Lean:
1. Forbid trumps permit (deny-overrides semantics)
2. Default deny (open-world assumption)
3. Explicit allow (no implicit grants)
4. Order independence (policy evaluation order doesn't matter)
5. Sound slicing (pruning preserves results)
6. Validation soundness (type-safe policies don't fail at runtime)
7. Termination

**Differential Random Testing (DRT)** — the practical workhorse:
- Tool: cargo-fuzz (libfuzzer-based, Rust)
- Platform: AWS ECS, 4 vCPUs / 8GB RAM per target
- Duration: 6-hour daily runs generating millions of test cases
- Lean model evaluation: 6µs median; Rust: 10µs median
- Two input generation modes: type-directed (schema → conforming policies/entities) and non-type-directed (ill-typed conditions, for error path coverage)

**Bug count**: 25 total (4 from formal proofs, 21 from DRT/PBT). The 4 proof-discovered bugs include a non-termination bug that testing alone couldn't find.

**Cost**: paper lists 12 authors from AWS, U Maryland, U Chicago. No dollar figures disclosed.

**What this means for Code Takeover**: Cedar's approach is the gold standard but requires:
- Formal model written separately in Lean (1,673 LOC for Cedar's core)
- Proof engineers (at least 4-5 based on author list)
- Continuous 6-hour fuzzing infrastructure
- Domain-specific input generators

Not directly adoptable for general code→spec→code workflows. The DRT insight is transferable: run both formal model and production code on the same random inputs, compare outputs.

**TLA+ Systematic Review (Nov 2024)** — arXiv:2411.13722 — decade of industrial TLA+ practice:

**Companies confirmed using TLA+**:
- Amazon Web Services (S3, DynamoDB) — the most referenced case (44% of papers cite Amazon's 2015 report)
- Microsoft (Cosmos DB)
- MongoDB
- Alibaba Group (PolarDB)
- Huawei Cloud (GaussDB/Taurus)
- Informal Systems (Tendermint blockchain)
- eBay, Audi, Thales, PharOS

**Usage patterns**:
```
Early design: 81% of cases
Debugging:    44%
Implementation: 38%

Language choice:
Pure TLA+:  50%
PlusCal:    19%
Both:       31%

Application domains:
Cloud applications:        63%
Railway control systems:   13%
Blockchain:                6%
```

**Only 19%** of papers address automated synchronization between specs and implementation code — the spec→code sync problem is almost universally treated as a manual process.

**Key challenges**: steep learning curve even with PlusCal; state space explosion when models are too detailed; the sweet spot is high-level design (early-stage), not line-level implementation verification.

**Dafny in Production (2025-2026)**:

*Dafny AI Assistant* (arXiv:2601.12845) — VS Code extension, 98.2% success rate for generating Dafny annotations (pre/postconditions, loop invariants) on 110 programs. Key finding: **proof helpers are the bottleneck** — they're what LLMs struggle with most. Spec generation without proof helpers (pre/post only) is solved at 96.4% logical equivalence to expert-written specs.

*Process*: generate-check-repair-minimize loop with Dafny verifier as oracle. Test cases with statically-checked assertions serve as compile-time oracles — the verifier checks that generated pre/postconditions allow test assertions to pass. Max 8 repair iterations, then minimization to reduce verbosity (raw repairs are 63% larger; minimized versions are only 10.6% larger than manual).

*LLMs compared*: Claude Opus 4.5 + GPT-5.2 multimodel = 98.2%. Single-model GPT-4 = substantially lower.

**Deductive Verification Adoption Survey (Oct 2025)** — arXiv:2510.20514 — 30 practitioner interviews (industry + academia). Three underexplored barriers:
- **Proof maintenance**: verified code requires re-verification after every change; cost compounds with codebase growth
- **Insufficient automation control**: users want to guide the prover, not submit to it
- **Usability gaps**: tools designed by theorem provers, not software engineers

Recommendations for tool builders: provide IDE integration, incremental verification, and explanation of counterexamples in plain language.

**Alloy**: limited industry adoption. Papers (arXiv:cs/0508109, arXiv:2503.05849) focus on improving usability rather than reporting production use. Primarily academic.

## 3. The Compression Problem

### 3.1 Information-Theoretic Foundations

**Kolmogorov complexity** provides the theoretical ceiling for spec compression. The key results:

**The uncomputability result**: the Kolmogorov complexity K(x) of any string x is not computable. For programs, K(program) = length of the shortest program that describes the same behavior. We cannot algorithmically find this shortest description — any spec compression tool is producing an approximation.

**For spec compression specifically**:
- The minimum spec for a program P is a string S such that some interpreter I produces behavior(P) from S
- K(behavior(P)) ≤ K(P) + constant (the spec cannot be longer than the code in information content)
- But K(behavior(P)) can be arbitrarily smaller than K(P) when code contains implementation-specific constants (variable names, algorithm choice, data layout)

**The compression limit**: a spec can compress a program only by eliminating *implementation redundancy* — choices that don't affect observable behavior. The irreducible information content of a program's behavior is the minimum spec length.

**Practical implication**: CodeSpeak's 6–10x compression ratios are achievable because most code *is* implementation redundancy. The 10x compressed spec is not lossy on behavior — it's lossy on implementation choices. This is the key insight that makes spec-as-source defensible.

### 3.2 What Dimensions Can Be Safely Compressed

Building on the K-complexity framing:

**Safely compressible (zero behavioral information)**:
- Variable/function names (observationally equivalent implementations differ only in identifier choice)
- Algorithm choice (merge sort vs quicksort — same output for same input)
- Data structure internals (array vs linked list — same interface)
- Error message wording (when not user-facing spec)
- Loop form (for vs while vs recursion)
- Import/module structure (reorganization doesn't affect behavior)

**Compressible with explicit spec** (must be stated if different from default):
- Performance characteristics (O(n²) vs O(n log n) — not observable through output, but observable through timing)
- Memory usage patterns (GC pressure, peak allocation)
- Error handling semantics (fail-fast vs accumulate — behavioral difference at boundary cases)
- Concurrency guarantees (thread safety — observable through race conditions)

**Non-compressible** (must preserve fully):
- All reachable outputs for all reachable inputs (the full I/O behavior)
- Side effect ordering (when multiple side effects exist)
- Exception types and messages (when they're part of the interface contract)
- Security properties (which inputs are rejected)

**The edge case problem as compression failure**: edge case behavior is the lowest-density information in a codebase — each edge case handles one narrow input region. But it's the highest-value information in a spec. A spec that compresses out edge case handling loses the 1% of behavior that users encounter only in production.

### 3.3 Measuring "Spec Completeness"

No formal "spec completeness ratio" metric exists in the literature. The closest proxies:

**Mutation score as behavioral coverage proxy**: a spec S is complete relative to test suite T if T achieves a mutation score of 1.0 — i.e., every mutant of the code is detected by T. This is impractical (equivalent mutants exist), but high mutation scores (>80%) indicate high spec fidelity.

**State field coverage** (arXiv:2510.03071): measures what proportion of fields modified by production code are verified by assertions. A spec with no assertions about internal state fields is incomplete relative to state behavior.

**Checked coverage** (Researchgate, Schuler et al.): an indicator distinguishing code that executes from code that is *asserted about*. Unchecked coverage = spec completeness gap.

**UCRBench's omission rate** as a proxy: when reverse-engineering specs from code, omission rate measures what fraction of behavior the LLM *fails to mention*. 49–64% omission at user-goal level → current LLMs spec at ~50% completeness for high-level behavior.

**Proposed completeness formula** (synthesized from above):
```
spec_completeness(S, P) =
  (mutation_score(tests_from_S, P) + checked_coverage(tests_from_S, P)) / 2
```
Where tests_from_S are tests generated from spec S run against program P.

## 4. Advanced Testing Approaches for Equivalence

### 4.1 Concolic Testing (Symbolic + Concrete)

**Tamarin: Concolic Disequivalence for MIPS** (arXiv:1801.02571) — the canonical tool for equivalence-via-concolic-execution:

*Technique: alternating concolic execution*:
1. Run both programs P1 and P2 concretely on random input
2. When outputs diverge → found disequivalence witness
3. When outputs agree → symbolically record path condition
4. Negate one path condition → generate new input exploring alternative path
5. Repeat until disequivalence found or input space exhausted

Integration: MIPS emulator with symbolic trace recording → Z3 SMT solver. The alternating approach is key: instead of exhaustively exploring P1 then P2, alternate between them to efficiently find divergence points.

**PASDA: Differential Symbolic Execution with Best-Effort Classification** (arXiv:2311.08071) — for the undecidable cases where formal proof fails:
- 141 non-/equivalent program pairs evaluated
- 61–74% correct classification depending on timeout (10s to 3600s)
- Outperforms 3 existing tools by 3–7%
- The "best effort" heuristic for undecided cases: apply partition-based approximation

**KLEE** (not a recent paper, but still state-of-art): symbolic execution engine for C/LLVM programs. Used extensively for finding bugs but less commonly for equivalence checking. Scalability limitations at >10,000 LOC still apply.

**Symbolic execution for COBOL→Java validation** (arXiv:2504.10548, arXiv:2506.10999):
- Symbolic execution generates unit tests for source COBOL programs with mocked external calls
- Tests transformed to equivalent JUnit tests for Java translation
- Semantic equivalence checked by running both suites
- 89.8% of C functions symbolically verified in RustAssure (arXiv:2510.07604); 69.9% showed equivalent symbolic return values

**Limitations of concolic for spec verification**:
- Path explosion for programs with loops and recursion
- Cannot handle programs with non-determinism (random, network, time)
- Z3 cannot solve all SMT formulas efficiently
- Best suited for pure functions with bounded inputs

### 4.2 Metamorphic Testing for Spec-Code Equivalence

**Metamorphic relations** (MRs) are properties of the form: "if input X produces output Y, then transformed input T(X) must produce predictable transformed output T(Y)." They solve the oracle problem by specifying *relationships between runs* rather than absolute expected outputs.

**CodeMetaAgent (Nov 2025)** — arXiv:2511.18249 — metamorphic relation-driven spec mutation for code generation:
- MRs applied proactively to *refine specifications* before generation, not just validate outputs
- Generates multiple specification variants that must all be satisfied consistently
- Results: +17% code generation accuracy, up to 99.81% code coverage gains (HumanEval-Pro, MBPP-Pro, SWE-Bench-Lite)

**Higher-order metamorphic relations for legal software** (arXiv:2509.13471):
- Tax code has no ground-truth oracle — cannot know the correct tax amount
- Higher-order MRs: compare system outputs across structured shifts among similar taxpayers
- "If taxpayer A with income X pays tax T, then taxpayer B with identical profile but income X+ΔX must pay approximately T+ΔT" (linearity MR)
- LLM-driven framework to auto-generate MRs from tax law statutes
- GPT-4o-mini: 45% pass rate vs GPT-4o/Claude 3.5: 9–15% (smaller model + systematic testing wins)

**Implication for Code Takeover**: when verifying spec→code correctness for domain logic, MRs derived from business rules are the appropriate oracle. Instead of testing absolute outputs (which requires knowing correct answers), test that the code satisfies domain invariants: "discount always < price", "refund ≤ payment amount", "tax_rate monotone in income bracket".

**Generating MRs automatically from specs**: the "LLM Assisted Coding with Metamorphic Specification Mutation Agent" approach (arXiv:2511.18249) suggests MRs can be generated from specs themselves. Workflow:
```
NL spec → identify invariant properties → generate metamorphic relations →
generate test variants → validate against implementation
```

### 4.3 Differential Fuzzing as Equivalence Oracle

**FLOURINE** (arXiv:2405.11514) — the most complete example of fuzzing-based cross-language equivalence checking:

*Technical components*:
- Fuzzer: libfuzzer + Bolero (Rust fuzzing framework)
- Serialization: program states to JSON for cross-language comparison
- Execution: foreign function interface (FFI) to run both C/Go and Rust
- Coverage: ~97% line coverage on average inputs
- Repair strategies tested: simple restart, hinted restart, base repair, CAPR (conversational repair)

*Results across 5 LLMs on 408 real-world functions*:
- GPT-4-Turbo: 47.7% (best)
- Claude 3 Sonnet: ~35%
- Claude 2.1, Gemini Pro, Mixtral: 21-33%

*Key finding*: counterexample-guided repair (providing failing inputs to LLM) improves by only 6–8%. Simple restart sometimes outperforms counterexample feedback because fuzzer-generated counterexamples are "unintuitive" for LLMs — random inputs that trigger bugs don't explain *why* the translation is wrong.

**VERT** (arXiv:2404.18852) — combines LLM + WebAssembly oracle for Rust transpilation:
- Oracle: compile source language to WebAssembly → compile Wasm to Rust → get canonical translation
- Verify: test LLM-generated Rust against canonical Rust using property-based testing
- Results: 31% → 54% pass rate (PBT) and 1% → 42% (bounded model-checking) when combining Claude-2 + VERT
- Limitation: requires a WebAssembly compiler for the source language

**Galapagos: N-Version Programming with LLMs** (arXiv:2408.09536) — automated generation of N functionally-equivalent program variants:
- LLMs generate multiple implementations of same spec
- Formal verification proves functional equivalence across variants
- Variants are "statically different after compilation, present diverging internal behavior at runtime" — diversity for fault tolerance
- Protected against real Clang miscompilation bugs
- Cross-language variants (different programming languages) can be formally proven equivalent

**The AFL/LibFuzzer connection**: AFL-style coverage-guided fuzzing is most effective when:
1. Input structure is known (can generate structured inputs)
2. Both programs have deterministic behavior
3. Side effects can be captured (not just return values)

For spec-code equivalence, the challenge is (3): specs often govern side effects (UI changes, database writes) that are hard to compare in a fuzzing framework.

### 4.4 Property-Based Testing with Hypothesis

**LLM-Generated PBT for Edge Cases** (arXiv:2510.25297) — analysis of property-based tests generated by Claude 4 Sonnet:

*PBT vs EBT detection rates on 16 HumanEval problems*:
- PBT alone: 68.75% bug detection
- EBT alone: 68.75% bug detection
- Combined PBT+EBT: **81.25%** bug detection

PBT strengths: performance bottlenecks, boundary condition violations through comprehensive input sampling
EBT strengths: specific boundary values, unusual input patterns

**The 12.5% that both miss**: bugs requiring both specific input structure AND knowledge of intended behavior — exactly the cases where a spec is needed as oracle.

**HardTests** (arXiv:2505.24098) — HARDTESTGEN pipeline:
- 47,000 problems with synthetically generated hard test cases
- +11.3pp precision, +17.5pp recall vs standard tests
- For hard problems: +40pp precision
- "Well-disguised wrong solutions" require edge cases that are difficult to synthesize automatically

**Boundary Value Analysis + LLMs** (arXiv:2501.14465) — LLMs for white-box BVA:
- LLMs can generate boundary value test inputs but "face challenges in certain areas, especially when handling complex or less common test inputs"
- Standard BVA fails the same: both struggle with implicit boundaries (business rules, state-dependent limits)

**TAROT curriculum for edge cases** (arXiv:2602.15449) — four-tier test hierarchy (basic → intermediate → complex → edge):
- Key insight: optimal curriculum is model-capability-dependent
- Less capable models: easy-to-hard progression works better
- More capable models: hard-first curriculum works better
- Applied to spec generation: start with simple behavioral properties, add complexity iteratively

## 5. The Edge Case Problem

### 5.1 LLM Edge Case Coverage: Empirical Data

**Mutation-Analysis of LLM Summaries** (arXiv:2602.17838) — reveals the edge case problem quantitatively:

*Experiment*: generate code summary → inject mutation → check if LLM updates summary to reflect mutated behavior

| Code type | Accuracy on mutated code |
|-----------|--------------------------|
| Single synthetic functions | 76.5% |
| Multi-threaded systems | **17.3%** |
| Human-written code (LBPP dataset) | 49.3% |
| GPT-5.2 on LBPP | 85.3% |

**"Describe algorithmic intent rather than actual mutated behavior"** — concrete failure: an LLM sees a sorting function with a mutated comparison operator. The model describes "sorts in ascending order" (pattern-recognized intent) without noticing that `<` was changed to `>` (actual mutated behavior). The spec is wrong for the mutated code.

**Implication**: LLM-generated specs describe *the typical case*, not *all cases*. Edge case behavior — exactly what mutations exercise — is what specs miss most.

### 5.2 The Ambiguity → Edge Case Bug Pipeline

**Bayesian requirements quality study** (arXiv:2401.01154) — causal analysis of ambiguity effects:
- 25 participants, 4 requirements documents with different defects
- **Ambiguous pronouns**: strong causal effect → incorrect associations in domain models
- **Passive voice**: minor impact only (counter to literature recommendations)
- Bayesian analysis reveals effect size that frequentist statistics misses

**Key finding**: pronoun ambiguity ("it processes the request and returns it") causally produces model errors. The pronoun refers to different entities in different reader's interpretations. This is the *semantic ambiguity* → *wrong implementation* pipeline.

**For spec-from-code**: the reverse problem — code has no pronouns, but it has implicit referents that a generated spec must make explicit. When code does `x = process(y); return x;`, the spec must state whether `x` is transformed `y` or a new object. Getting this wrong produces an edge case bug in regenerated code.

**Spec ambiguity taxonomy** (synthesized):
1. **Referential ambiguity**: unclear what noun an operation acts on
2. **Quantifier ambiguity**: "all users" vs "active users" vs "any user"
3. **Temporal ambiguity**: "before" — immediately before? Eventually before?
4. **Scope ambiguity**: does a constraint apply to one module or all modules?
5. **Default ambiguity**: what happens when a required field is missing?

Each ambiguity type maps to a class of edge case bugs in generated code.

### 5.3 Enumerating Edge Cases Automatically from Specs

**STALAGMITE approach (grammar inference)**: extract the grammar implicit in parser code → enumerate corner cases of the grammar. For a JSON parser, the implicit grammar includes: empty object, deeply nested objects, duplicate keys, unicode edge cases, number precision limits. The grammar-derived edge cases are exhaustive within the grammar's structure.

**Metamorphic relation generation from specs**: CodeMetaAgent (arXiv:2511.18249) generates MRs from spec text. Each MR defines a class of edge cases: if the spec says "discount applies to all items", the MR is "order with N items has N times more discount than order with 1 item". This generates structured edge cases from spec properties.

**HardTests pipeline** for edge case synthesis:
```
1. Extract known correct solutions as ground truth
2. Generate adversarial inputs designed to distinguish correct from incorrect
3. Filter to "hard" inputs where subtly wrong solutions pass standard tests
```
17.5pp recall improvement suggests this catches real edge cases that simple test generation misses.

**LLM corner case generation prompt patterns** (from CodeMetaAgent):
```
Given specification: [S]
Generate edge cases by:
1. Identify boundary conditions in spec (limits, empty, single element)
2. Identify interaction between conditions (when multiple rules apply simultaneously)
3. Identify undefined/implicit cases (what spec doesn't say)
4. Generate inputs that exercise each edge case class
```
The third category is the hardest: what the spec *doesn't say* is where real-world bugs live.

## 6. Incremental Conversion Strategies

### 6.1 File-by-File vs Module-by-Module vs Feature-by-Feature

**TimeMachine-bench (2026)** — arXiv:2601.22597 — key failure mode reveals the ordering problem: LLMs "exploit low test coverage" to generate spurious solutions that pass tests without actually solving the migration. This is the incremental conversion risk: if you convert file A before its dependencies, the test suite for A may not cover behaviors that depend on B, C.

**Optimal ordering heuristics** (synthesized from migration papers):

1. **Dependency-first**: convert leaves of the dependency graph first (utility functions, no dependencies). Convert core modules last.
   - Benefit: converts are self-contained, tests fully exercise the converted unit
   - Risk: core business logic changes invalidate leaf specs

2. **Test-coverage-first**: rank modules by test coverage, convert highest-coverage first. Converts in low-coverage zones must be deferred until tests are written.
   - Benefit: high-confidence conversions first
   - Risk: may defer most important business logic (often undertested)

3. **Risk-score-last**: define risk as `risk(module) = (business_criticality × (1 - test_coverage))`. Convert low-risk first.
   - Benefit: builds confidence before tackling critical code
   - Practical implementation: `git log --follow -p <file>` change frequency as criticality proxy; existing test coverage from pytest-cov/JaCoCo

**Feature-by-feature alternative**: instead of file boundaries, use feature boundaries:
- Identify all files that implement Feature X
- Convert Feature X as a unit, keeping the feature's test suite as oracle
- Repeat for Feature Y

This approach aligns with Testora's NL oracle: the PR description is a feature-level spec, so the test oracle should be feature-level. File-level conversion with file-level tests misses cross-file feature behaviors.

**Google's Kythe approach** (arXiv:2504.09691) does neither — it migrates by *change location* (all places an ID field is referenced), which is a semantic unit rather than a syntactic file boundary. This is effectively the feature-by-feature approach but defined by ID-field scope rather than user story.

### 6.2 The Strangler Fig Pattern for Spec Adoption

The strangler fig pattern (Fowler) for microservices migration applies directly to spec adoption:

```
Phase 1: Identify facade interface
  → Write spec for the facade (inputs/outputs only)
  → Don't convert internals yet

Phase 2: New code goes spec-first
  → All new modules start from spec
  → Legacy modules still code-first

Phase 3: Incremental spec extraction
  → Extract specs from legacy code one module at a time
  → Old code remains running during spec extraction

Phase 4: Validate and cut over
  → Run old and spec-generated code in parallel (differential testing)
  → Cut over after equivalence confidence threshold

Phase 5: Legacy module retirement
  → Remove old code after N days of parallel success
```

No paper directly studies this applied to spec adoption. The closest is FLOURINE's LLM-in-the-loop approach: old code and new Rust translation run in parallel with differential fuzzing comparing outputs. If equivalent for sufficient inputs, translation is accepted.

**AIMigrate's diff approach** generalizes: when the "spec" is a migration guide (library diff), providing the diff dramatically outperforms providing the full code. This suggests that incremental migration with a well-structured change log enables better LLM performance than big-bang migration.

### 6.3 Risk Scoring for Conversion Readiness

No paper specifically defines a "conversion readiness score" for spec adoption, but the components are identifiable:

```python
def conversion_readiness(module):
    """
    Returns 0-1 score, higher = safer to convert first.
    """
    return (
        test_coverage_ratio(module)           # 0-1: more tests = safer
        * (1 - change_frequency_normalized(module))  # less churn = more stable spec
        * (1 - cyclomatic_complexity_normalized(module))  # simpler = more spec-able
        * interface_stability(module)         # stable interface = reliable spec
    )
```

**Operationalizing each factor**:
- `test_coverage_ratio`: line/branch coverage from existing test suite
- `change_frequency_normalized`: `git log --since="1 year" --oneline <path> | wc -l` / max_changes
- `cyclomatic_complexity`: from radon (Python) or SonarQube
- `interface_stability`: count of callers * times caller signatures changed in 6 months

High-readiness modules: utility functions, data transformers, schema validators.
Low-readiness modules: core business logic with complex state, frequently changed features, modules with <50% test coverage.

### 6.4 Rollback Strategies

**KCoEvo approach** (arXiv:2603.07581) — knowledge graph-based migration tracks API evolution paths:
```
API v1 → API v2 → API v3
```
Each node in the knowledge graph stores the code transformation pattern. If a v3 migration fails, rollback traverses the graph to v2. The knowledge graph IS the rollback plan.

**COBOL→Java dual-running** (arXiv:2504.10548): run both COBOL and Java simultaneously in production (shadow mode). If Java output diverges from COBOL, route back to COBOL automatically. Cutover happens when shadow mode shows 99%+ agreement over N production requests.

**Git-based rollback for spec adoption**: keep the pre-conversion code in a separate branch. When differential testing detects equivalence failure in production, the spec system automatically reverts to the old code while filing an issue. The spec itself becomes the "issue description" for the regression.

## 7. Multi-Language Considerations

### 7.1 Cross-Language Behavioral Equivalence Testing

The most developed area is C/Go → Rust translation, motivated by memory safety:

**FLOURINE** (arXiv:2405.11514) — complete cross-language equivalence via differential fuzzing. libfuzzer generates inputs, both implementations execute via FFI, outputs serialized to JSON and compared. 47% success rate at I/O equivalence.

**RustAssure** (arXiv:2510.07604) — differential *symbolic* testing for C→Rust:
- Symbolic execution on both C and Rust
- Compare symbolic return values (symbolic equivalence, not just concrete I/O)
- 89.8% compilation success, 69.9% with equivalent symbolic return values

**VERT** (arXiv:2404.18852) — WebAssembly oracle approach:
- Source language → Wasm → canonical Rust (oracle)
- LLM-generated Rust verified against oracle via PBT
- 54% PBT pass rate, 42% bounded model-checking pass rate

**BootTrans** (arXiv:2601.03512) — bootstrapping for multi-language translation:
- Key insight: test suites are "functionally invariant and cross-lingually portable"
- Same JUnit-style tests can verify Python, Java, TypeScript implementations of the same function
- Reinforcement learning: use cross-lingual test execution as reward signal
- Language-aware weighting: prioritize harder translation directions during training

**COBOL→Java validation** (arXiv:2504.10548, arXiv:2506.10999):
- Symbolic execution generates COBOL tests with mocked external calls
- Transform to JUnit equivalents (same mocking structure)
- Both test suites validate semantic equivalence

### 7.2 Programming Language Families and Spec Transfer

**Language Family Research** (arXiv:2512.19509) — 19 programming languages clustered by 21 linguistic features:

Key clusters identified:
- Family 1: C, C++, Java, Swift (static typing, C-family syntax, memory management)
- Go: central language, highest cross-language similarity to all others
- Python/Ruby: dynamic typing, expressive syntax
- Haskell/Erlang: functional paradigm, different from all C-family

**Implications for spec generation in polyglot systems**:

1. **Within-family spec transfer**: a spec written for a Java module can be applied to generate equivalent C++ with minimal adaptation. The 21 linguistic features that make them similar (variable definition, control structures, method declarations) map directly.

2. **Across-family spec gaps**: a Python spec may not generate idiomatic Java. Python specs often rely on duck typing ("object with a `process()` method") while Java specs require explicit interface definitions (`implements Processable`). The spec must encode the interface differently per target language family.

3. **Go as spec hub**: Go's central position suggests it could serve as an intermediate spec language. "Spec in Go style" → translate to C++, Python, Rust. This aligns with Go's design philosophy (explicit over implicit, one obvious way to do things).

**LANTERN** (arXiv:2503.22512) — cross-language repair leverages LLM proficiency differentials:
- LLMs are better at repairing Python bugs than Rust bugs (training data imbalance)
- Translate Rust buggy code → Python → fix in Python → translate back to Rust
- Rust: +22.09% improvement in Pass@10
- Principle: route spec generation through the language where LLMs are most competent

**MultiOOP benchmark** (arXiv:2509.26111) — six languages (Python, PHP, C++, C#, Java, JavaScript) for OOP code generation. Shows LLM capability varies substantially: Java and Python are the strongest; C++ and PHP are weakest. A polyglot spec system should generate Java/Python first, then adapt.

### 7.3 How Spec Language Differs for Different Target Languages

No paper directly addresses "spec dialect per target language", but the evidence points to three dimensions:

**Type expressiveness**:
- For Python targets: specs can use duck typing ("accepts any object with `len()` and `__getitem__()`")
- For Java/C++ targets: specs must use nominal types ("accepts `Sequence<T>` where T implements `Comparable<T>`")
- For Rust targets: specs must include lifetime and borrowing constraints ("borrows `data` for duration of `self`")

**Concurrency model**:
- For Go: spec should express channel-based communication ("sends result on channel when complete")
- For Java: spec should express monitor-based synchronization ("synchronized on `this`")
- For Rust: spec should express ownership transfer ("takes ownership of input")

**Error handling model**:
- For Python: spec expresses exceptions ("raises `ValueError` if input is empty")
- For Rust: spec expresses Result types ("returns `Err(ParseError)` on invalid input")
- For Go: spec expresses multiple return values ("returns `(T, error)`, error non-nil on failure")

**DSL constraint approach** (Anka, arXiv:2512.23214) suggests the solution: define a spec DSL with explicit constructs for each target language's semantic model. The spec is then *compiled* to language-appropriate code patterns, not just summarized.

### 7.4 Cross-Language Equivalence: The Polyglot Codebase Challenge

A Python API server + Go backend + TypeScript frontend is a common production architecture. Behavioral equivalence across language boundaries is under-researched but practically important.

**Interface-boundary approach** (synthesized):
- Define equivalence at API boundary, not implementation level
- HTTP endpoints have defined input/output schema (OpenAPI)
- Each language implementation must satisfy the same JSON schema
- Differential testing: send same inputs to both old and new implementations via HTTP, compare responses

This degrades gracefully: you don't need language-level equivalence checking if you can test at the network boundary. The spec is OpenAPI (or gRPC proto), and the oracle is the API responses.

**The challenge**: internal state that affects future responses. A refactored database schema might produce identical responses for 99% of inputs but diverge when querying data written under the old schema. Cross-language equivalence checking must model state effects.

## Open Problems Not Addressed by Any Paper

1. **Spec completeness quantification**: no standard metric for "what percentage of observable behavior is captured by a spec". Mutation score is the proxy, but requires running both code and tests.

2. **Cross-language spec transfer**: no tool that takes a spec for a Python module and generates equivalent Go code while preserving all behavioral guarantees. The language family clustering (arXiv:2512.19509) provides the theoretical basis; the tooling doesn't exist.

3. **Concurrency spec extraction**: mutation analysis accuracy drops from 76.5% to 17.3% for multi-threaded code. No solution for extracting behavioral specs of concurrent systems. Even formal tools (Lean, Dafny) handle concurrency poorly at scale.

4. **Edge case completeness from specs**: even with MRs auto-generated from specs, the 12.5% that both PBT and EBT miss requires knowing what the spec *doesn't say*. No systematic method for enumerating implicit spec requirements.

5. **Environment co-migration**: code specs don't capture infrastructure dependencies (env vars, config files, network endpoints). Migrating code without co-migrating environment causes 20-30% of migration failures (implied by arXiv:2602.09944 finding). No spec format captures both.

6. **Invariant completeness for dynamic systems**: ClassInvGen (arXiv:2502.18917) generates class invariants but doesn't guarantee they're *complete* — a generated invariant might describe typical behavior while missing invariants that only apply at edge states. The "correctness and completeness" testing harness is mentioned but not detailed.
