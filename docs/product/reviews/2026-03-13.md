## What We Learned

- the git adapter becomes much more credible once it points at stored artifacts instead of only automated tests
- a checked-in sandbox repo is a reasonable bridge when the top-level workspace is not itself a git checkout
- the wedge still reads cleanly when the proof fixture stays tiny and path-compatible with the main task inputs
- crossing into another Go module is simpler through a built binary than through `go run`
- the external repo proof is materially stronger than the sandbox proof, even though the repo is still small
- historical third-party commits are a better signal than our own authored external changes
- commit-message-derived boundaries are imperfect, but still useful enough to expose obvious drift
- maintainer review language is enough to tighten a usable file boundary without repo-specific heuristics
- the same PR can serve as both `inspect` and `approve` proof when discussion visibly narrows the change
- `raw diff` versus `review object` is a real product advantage, not only a nicer artifact format
- issue text alone can be enough to derive a useful bounded review object when the merged change is clean
- the wedge proof now spans the main task-truth sources we care about: task packet, repo history, PR discussion, and issue text
- the first useful productization step is path compression, not smarter inference
- shell-facing path patterns need to be quoted or the shell will silently expand them before Specpunk sees them
