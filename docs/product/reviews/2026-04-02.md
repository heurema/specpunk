## What We Learned

- the old self-hosting ceiling was architectural, not prompt-level; execution lanes and controller-owned mutation were the real fix
- no-shell patch generation is safer and more reliable once the controller supplies the right local excerpts
- contract drafting quality depends on policy and entry-point inference, not only file-token heuristics
- zero-config is only convincing once it works in an isolated temp environment, not just in unit tests
- operator commands become useful when they end in directives or actions, not when they only dump state
- grouped review surfaces (`triage`, `pipeline`) are much easier to act on than flat tables
- provenance for `ask` is more trustworthy when emitted by the CLI from structured local state instead of delegated to the model

## What Is Now Wrong In The Docs

- roadmap phrasing for Phase 2 still understates how much of the operator workflow is already product-shaped in nested `punk`
- the docs do not yet clearly distinguish between `status`, `doctor`, `morning`, `triage`, `ask`, and `pipeline` as separate operator surfaces
- there is no short milestone note that says the bounded existing-file glue slice ceiling has been practically broken

## What We Cut, Moved, Or Reduced

- cut further blind tuning of prompt/retry seeds once it became clear the problem was execution architecture
- moved resolver ambiguity handling to manual bounded implementation after fail-closed self-hosting correctly refused unsafe patching
- reduced several reliability follow-ups to the minimum controller-side fixes needed to validate the next real product slice
